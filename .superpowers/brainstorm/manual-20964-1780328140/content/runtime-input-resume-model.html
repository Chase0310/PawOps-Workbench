技术选型: Waiting-for-input 恢复模型

生命周期已经包含 waiting-for-input。现在决定外部输入回来后，run 如何恢复执行。

A

Raw Input Forward

外部直接把文本输入转发给 executor。实现最快，但 runtime 很难理解输入语义。

B

Structured Input Resume

runtime 接收结构化输入，如 reply、approve、reject、provide-secret，再恢复 run。

C

Executor-owned Dialog

输入对话几乎完全由 executor 自己维护，runtime 只转发消息。最灵活，但 runtime 会失去统一交互语义。

推荐: B
run enters waiting-for-input with inputRequest metadata
UI renders expected input type and prompt
runtime.submitInput(runId, input)
runtime validates input shape, emits resume event, hands payload to executor

Why B fits

  • Workbench 可以理解“这次在等什么输入”。
  • 以后不止 CLI executor 时，交互语义还能统一。
  • 审批、补充参数、补 secret 都能走同一模型。

Accepted cost

  • 第一版要先定义少量输入类型。
  • 不能把输入协议设计得过大，否则会反向绑死 executor。
