技术选型: Runtime 控制面

事件流已经定为结构化事件。现在决定外部如何驱动 runtime。

A

Imperative Method API

通过方法调用控制 runtime，例如 startRun、cancelRun、subscribe。

B

Command Bus

外部把 start、cancel、resume 都作为命令投递给 runtime，由内部命令总线执行。

C

State Machine First

外部只做状态变更请求，runtime 按状态机推进执行，适合更重的后台系统。

推荐: A with explicit run handles
createRuntime(services, profiles)
runtime.startRun(input) -> runHandle
runHandle.subscribe(listener)
runHandle.cancel() / pause() / resume()
runtime.listRuns() / getRun(runId)

Why A fits now

  • 最适合嵌入式 package 形态。
  • Electron 和 Web debug wrapper 都容易包一层 API。
  • 先做清晰对象模型，后面仍可映射到 HTTP 或 command bus。

Accepted cost

  • 分布式和多进程场景下需要再包一层 transport。
  • 如果一开始就追求强状态机，A 不如 C 那么“后端化”。
