技术选型: CLI Adapter 统一事件粒度

适配器已经确定为 session-oriented。现在决定统一事件要压到多粗，才能既够用又不泄漏太多底层差异。

A

Raw Stream Events

几乎直接转发 stdout、stderr、exit。实现快，但 runtime 和 UI 要理解太多底层差异。

B

Normalized Session Events

统一成 started、output.chunk、input.requested、status、completed、failed、interrupted 这类会话事件。

C

Semantic High-level Events

适配器尝试直接产出 plan.updated、file.edited、task.finished 这类高语义事件。表达强，但误判风险高。

推荐: B
started
output.chunk { stream, text }
input.requested { kind, prompt? }
status.changed { status, detail? }
warning { code, message }
completed { summary?, exitCode? }
failed { reason, exitCode? }
interrupted { mode: soft | hard }

Why B fits

  • 对 Workbench 和 runtime 足够稳定，不依赖某家 CLI 的原始输出格式。
  • 仍然保留流式输出和等待输入这些关键交互。
  • 不会过早假设 CLI 能可靠提供高语义结构。

Accepted cost

  • 部分 CLI 特有细节会被折叠掉。
  • 如果后面真要做高语义分析，需要在更上层二次派生。
