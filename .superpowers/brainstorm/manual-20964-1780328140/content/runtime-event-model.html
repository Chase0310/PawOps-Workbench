技术选型: Runtime 事件模型

Workbench、日志、trace、Web 调试都要消费同一套事件。先定事件边界，再定 API。

A

Log Lines First

运行时只吐文本日志和少量状态更新。实现快，但 UI 和 trace 需要二次解析。

B

Typed Domain Events

运行时输出结构化事件，如 run started、step started、tool called、CLI chunk、completed。

C

OpenTelemetry First

直接以 span 和 telemetry 为核心。对分析强，但不适合作为产品 UI 的主事件总线。

推荐: B + OTel Bridge Later
Runtime emits typed domain events
Workbench subscribes to event stream for UI
Log service derives human-readable logs from events
Trace service converts selected events into spans later

Why B fits

  • UI 不需要解析日志文本。
  • CLI adapter 和 MCP adapter 都能接入同一事件管道。
  • 后续加 trace、replay、persist 都更自然。

Accepted cost

  • 前期要先设计事件 schema。
  • 需要克制事件种类，避免过度建模。
