技术选型: 哪些 CLI 细节要归一化，哪些保留为 metadata

适配层负责吸收 Codex CLI 和 Claude Code CLI 的差异，但不是所有差异都应该提升成统一契约。

A

Normalize Almost Everything

尽量把 CLI 差异都提成统一字段。上层最简单，但适配层会越来越重，也容易误抽象。

B

Normalize Critical, Keep The Rest as Metadata

只统一 run 控制和交互关键点，CLI 特有细节保留在 metadata 里，供调试和高级能力使用。

C

Metadata First

大部分都先原样保留为 metadata，只统一极薄控制面。实现快，但上层长期会越来越脏。

推荐: B
normalize these:
started / completed / failed / interrupted
output.chunk
input.requested
status.changed
exit semantics
keep as metadata:
raw CLI event names
provider-specific warnings
CLI version / binary info
adapter-specific diagnostics

Why B fits

  • 上层只依赖真正稳定、跨 CLI 的行为语义。
  • 调试时仍能看到 CLI 原始细节，不会被适配层吞掉。
  • 避免为了统一而发明虚假的通用概念。

Accepted cost

  • UI 如果想展示某些 CLI 特有能力，需要读 metadata。
  • 要明确 metadata 只是补充信息，不能反向成为主契约。
