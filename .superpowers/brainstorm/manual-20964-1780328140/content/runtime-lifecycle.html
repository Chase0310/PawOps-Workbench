技术选型: Run 生命周期

控制面已定为 Method API。现在决定一个 run 从创建到结束要经历哪些阶段。

A

Minimal Lifecycle

created -> running -> completed | failed | cancelled。先保证最小闭环。

B

Interactive Lifecycle

created -> running -> waiting-for-input | paused -> completed | failed | cancelled。

C

Durable Lifecycle

queued、resuming、retrying、timed-out、orphaned 都先建模，偏后台控制平面。

推荐: B
created
running
waiting-for-input or paused
completed / failed / cancelled

Why B fits

  • coding CLI 和后续人机交互会需要等待输入。
  • 比 A 更贴近真实工作流，但没有 C 那么重。
  • 桌面工作台更容易呈现运行中的状态。

Accepted cost

  • 状态管理比最小闭环复杂一点。
  • 要区分 paused 和 waiting-for-input 的语义。
