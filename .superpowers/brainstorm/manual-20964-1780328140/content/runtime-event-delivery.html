技术选型: Runtime 事件投递接口

我们已经确认 runtime 会发结构化事件。现在决定调用方怎么消费这些事件。

A

Callback Subscribe

通过 subscribe(listener) 接收事件。最直观，最适合桌面嵌入和 Web wrapper。

B

AsyncIterable Stream

通过 for await 消费事件流。组合性强，但前端和 UI 层接入不如 callback 直接。

C

Both Forms

同时提供 subscribe(listener) 和事件流接口。能力完整，但第一版 API 面会更大。

推荐: A first
runHandle.subscribe(listener)
listener receives typed RuntimeEvent
desktop UI and debug server can forward events easily
later: derive AsyncIterable if real demand appears

Why A fits

  • 和 Method API 风格一致。
  • Electron、WebSocket、SSE 包装都很顺。
  • 比一次性暴露多种消费范式更克制。

Accepted cost

  • 流式组合能力不如 AsyncIterable 自然。
  • 后续如果需要复杂管道，可能再补第二种消费接口。
