技术选型: Runtime 执行引擎抽象

外围控制面已经清楚了。现在决定一个 run 内部是如何真正执行任务的。

A

CLI Executor First

第一版 runtime 只支持外部 coding CLI 执行器，例如 Codex、Claude Code、Gemini CLI。

B

Executor Interface

runtime 面向统一执行器接口，第一版先实现 CLI executor，后续可以加 model loop executor。

C

Dual Engine from Day 1

第一版同时支持 CLI executor 和内建 LLM tool-loop executor，能力最完整但复杂度最高。

推荐: B
runtime-core orchestrates runs and events
executor interface defines start, input, interrupt, cleanup
v1 implementation: coding CLI executor
future implementation: model loop executor, background executor

Why B fits

  • 先跑通 coding CLI，但不把 runtime 写死成“只是 CLI wrapper”。
  • 和 agent profile 的未来扩展更匹配。
  • 后续加内建 executor 时，不需要重写 runtime 控制面和事件模型。

Accepted cost

  • 第一版要先定义执行器接口。
  • 抽象过头会有风险，需要保持接口足够小。
