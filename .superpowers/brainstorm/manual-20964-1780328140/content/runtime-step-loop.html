技术选型: Runtime 执行循环

控制面、生命周期、事件模型都定了。现在决定 run 在 running 状态下如何推进 step。

A

Synchronous Step Loop

startRun 后内部 while-loop 顺序推进，直到完成或失败。实现最简单，但中断和等待输入的处理会偏硬。

B

Cooperative Step Loop

每个 step 在边界处检查 cancel、pause、waiting-for-input，再决定下一步。适合本地 runtime 和交互式工作流。

C

Queued Actor Loop

每个 run 像 actor 一样靠消息推进。并发和恢复能力更强，但第一版明显过重。

推荐: B
step start -> emit event
execute tool or CLI work
checkpoint state
check cancel / pause / waiting-for-input
advance to next step or terminal state

Why B fits

  • 和我们选的交互式 lifecycle 一致。
  • 适合 coding CLI 这种会流式输出、可能等待输入的执行器。
  • 比纯同步循环更容易支持 cancel 和 pause。

Accepted cost

  • 比最简单的 while-loop 多一层 step 边界约束。
  • 需要先定义什么算一个 step。
