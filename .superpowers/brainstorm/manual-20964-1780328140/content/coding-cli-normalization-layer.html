技术选型: CLI 差异归一化在哪一层处理

launch(context) 已经收敛。现在决定 Codex CLI 和 Claude Code CLI 的行为差异应该在哪一层被吸收。

A

Runtime Absorbs Differences

adapter 只做最薄封装，差异主要由 runtime 理解和处理。实现快，但 runtime 会被 CLI 细节污染。

B

Adapter Normalizes

每个 adapter 内部吸收启动参数、输出格式、输入等待和中断差异，对上统一 contract。

C

External Compat Layer

单独做一层 CLI compatibility layer，再由 adapter 调用。长期可扩展，但第一版层次偏多。

推荐: B
runtime only sees unified session contract
codex adapter handles Codex-specific launch and output quirks
claude adapter handles Claude-specific prompt/input/interrupt quirks
future adapters follow the same normalization boundary

Why B fits

  • 保护 runtime-core 不被具体 CLI 污染。
  • 统一 contract 的价值真正落在 adapter 层。
  • 后续加第三个 CLI 时，不需要改 runtime 语义。

Accepted cost

  • adapter 实现会比简单 process wrapper 更厚一点。
  • 需要定义什么差异必须归一化，什么可以保留为 metadata。
