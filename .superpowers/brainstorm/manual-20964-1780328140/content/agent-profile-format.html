技术选型: Agent Profile 定义格式

Runtime core 的主骨架已基本收敛。现在决定每只狗狗 agent 怎么被定义和加载。

A

TypeScript Profile Module

每个 agent 用 TS 文件导出 profile。表达力最强，但配置和代码耦合更深。

B

Declarative Manifest

每个 agent 用 YAML 或 JSON manifest 描述名称、prompt、skills、MCP、executor、权限、预算。

C

Hybrid: Manifest + Optional Hooks

主定义走声明式 manifest，少量高级行为可挂可选 TS hook。兼顾可配置性和扩展性。

推荐: C
profile manifest owns stable fields: id, name, role, executor, skills, MCP, permissions, budget
runtime validates manifest with schema at load time
advanced behavior uses optional hook modules only when truly needed
most agents stay data-driven and easy to inspect

Why C fits

  • 大多数 agent 可以纯配置化定义，方便扩展和审查。
  • 不会被纯声明式格式卡死，少数复杂 agent 仍可扩展。
  • 比“全部写成 TS 模块”更适合做目录、市场和共享模板。

Accepted cost

  • 需要定义 manifest schema 和 hook 边界。
  • 要克制 hook 能力，避免重新退化成全代码配置。
