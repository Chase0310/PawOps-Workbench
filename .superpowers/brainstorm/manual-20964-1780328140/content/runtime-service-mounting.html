技术选型: Runtime 服务挂载方式

控制面和生命周期都已收敛。现在决定 memory、log、trace、tool、MCP 等服务怎么挂到 runtime 上。

A

Constructor Dependencies

创建 runtime 时一次性传入 services 和 registries。最直接，最适合第一版。

B

Plugin System First

所有能力都通过插件注册和 hook 挂载。扩展强，但第一版抽象会偏重。

C

Service Container

先做一个 DI 容器或 registry，通过 token 解析依赖。适合大系统，但前期复杂。

推荐: A with explicit extension points
createRuntime({ store, eventBus, logger, tracer, toolRegistry, mcpRegistry, profileRegistry })
core only depends on interfaces
later: plugin layer can wrap these extension points

Why A fits

  • 接口边界清楚，最容易测试。
  • 和嵌入式 package 形态天然匹配。
  • 不会过早引入插件系统复杂度。

Accepted cost

  • 后续如果扩展点很多，需要再抽出 plugin 层。
  • 构造参数需要保持克制，避免膨胀。
