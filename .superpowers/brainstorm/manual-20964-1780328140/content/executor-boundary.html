技术选型: Runtime 与 Executor 的职责边界

已经确认用统一执行器接口。现在决定执行器到底是“黑盒过程”还是“受 runtime 驱动的协作单元”。

A

Opaque Executor

executor 像黑盒任务，runtime 只负责启动、收输出、取消和结束。最容易兼容外部 CLI。

B

Cooperative Executor

runtime 负责 run 生命周期、事件、状态；executor 负责实际执行，并在 step 边界上配合 pause、input、interrupt。

C

Autonomous Executor

executor 自己维护主要状态机，runtime 更像外层壳。适合复杂引擎，但会削弱 runtime-core 的中心地位。

推荐: B
Runtime owns run state, events, persistence, cancellation policy
Executor owns concrete execution mechanics
Executor reports output, waits-for-input, completion, error
Runtime translates these into domain events and lifecycle transitions

Why B fits

  • 不把 runtime 降级成简单的 process wrapper。
  • 也不强迫 CLI executor 暴露过多内部细节。
  • 以后新增 model-loop executor 时，边界还能复用。

Accepted cost

  • 需要认真设计 executor 回调和状态上报接口。
  • step 边界的定义必须保持克制。
